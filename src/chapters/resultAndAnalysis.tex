\chapter{RESULTS AND ANALYSIS}
The following report details the training and evaluation of a \gls{lbc} architecture for \gls{hpd}. The model was trained over 45 epochs using a \textit{OneCycleLR} scheduler before early stopping was triggered.

\section{MobileNetV2 Architecture Performance}
The utilization of a MobileNetV2 architecture for the task of \gls{hpd} demonstrated
significantly higher accuracy compared to the lightweight \gls{lbcnn} baseline, at the cost of
increased model complexity. The model was trained for 12 epochs before early stopping was
triggered, using a \textit{OneCycleLR} scheduler with a maximum learning rate of $0.01$.

\section{Overall Comparative Analysis}

The experimental results summarised in \cref{tab:model_comparison} show that the MobileNetV2 outperforms the LBCNN in all the evaluation metrics with a clear edge in performance. The MobileNetV2 achieved $86.18\%$ test accuracy and $0.931\%$ gls{auc}, while the LBCNN achieved $71.74\%$ and $0.777\%$, respectively, which represents an improvement of $14.44\%$ percentage points and $0.154\%$. This gap is present for all the per-class metrics, with MobileNetV2 being better than gls{lbcnn} in terms of both precision and recall for both classes.

The difference in training behaviour is equally clear. The gls{lbcnn} loss and accuracy curves are very volatile for all the 45 epochs with the validation accuracy jumping sharply between successive epochs. The instability is a direct result of the very small training set (737 samples) where individual batches have an outsized impact on the gradient signal. The MobileNetV2 model trained on 18,432 samples with ImageNet pretrained weights converges smoothly in 12 epochs, with the loss and accuracy curves declining and increasing monotonically, respectively. The pretrained initialisation removes the need to learn low level features from scratch, and the larger dataset provides stable gradients together explaining why MobileNetV2 converges faster, more cleanly and to a significantly better optimum..

This picture is supported by the roc and precision–recall curves. The roc curve of MobileNetV2 goes up sharply towards the top left corner, suggesting high true positive rates for low false positive rates and demonstrating that the class probability scores are well separated. The \gls{lbcnn} ROC curve is more gradual, reflecting that its 64-dimensional binary feature representations achieve less class separability. Similarly, MobileNetV2 achieves higher precision over a wider range of recall thresholds. \gls{lbcnn} precision drops more quickly with increasing recall, in line with a model working close to the limits of its representational capacity.

However, this comparison should be understood in the context of the fundamentally different constraints under which each model operates. MobileNetV2 has 2,225,858 parameters and takes up 8.49~MB in FP32, which is way beyond the memory budget of any commodity \gls{mcu}. The \gls{lbcnn}, which has only 19,779 parameters and a deployed size of just 0.02 MB after INT8 quantization, easily fits within the ESP32-S3’s 512~KB \gls{sram}. The 14.44 percentage point accuracy trade-off is thus not a shortcoming but an intentional result of designing for deployment on hardware that is about \textbf{425$\times$ more memory-constrained} than what MobileNetV2 requires. For real-time, battery-powered, GPU-free edge inference, \gls{lbcnn} offers a principled and practical solution to the accuracy-efficiency trade-off of \gls{tinyml} deployments.

\section{System Overview and Evaluation Methodology}
 
We evaluate this distributed inference pipeline for four different processing stages: frame capture and tiling at the \gls{esp}-CAM, tile transmission from the \gls{esp}-CAM to the two ESP32-S3 worker nodes, tile-level feature extraction and attention scoring at each worker node, weighted feature transmission from the worker nodes to the ESP32 aggregator, and final classification at the aggregator. Each stage was independently benchmarked against dedicated test sketches with synthetic inputs to isolate stage specific latency from end-to-end confounding factors. All the timing measurements were done using the ESP32 hardware \textit{micros()} counter, with the CPU clock set to 240MHz. \Cref{tab:pipeline_latency} presents the complete latency breakdown across

\subsection{ESP-CAM: Frame Capture and Tiling}
 
The \gls{esp}-CAM node captures a 224$\times$224 gray-scale frame and splits it into nine overlapping 96$\times$96 tiles with a step size of 72 pixels (25\% overlap) along both spatial axes. The whole capture-to-tile pipeline takes 4~ms, rendering this stage completely insignificant compared to all downstream processing. This confirms the fact that the tiling strategy itself incurs no meaningful overhead, and that the \gls{esp}-CAM is able to achieve the target 1~\gls{fps} frame rate from a capture perspective alone.
 
The 4~ms figure encompasses camera sensor readout, RGB565-to-grayscale conversion, and
the nine sequential tile extraction operations. At this rate, the \gls{esp}-CAM could
theoretically support up to 250~\gls{fps} in isolation, meaning the capture stage will
never be the bottleneck regardless of how the downstream pipeline is optimised.
 
\subsection{Tile Transmission: ESP-CAM to Worker Nodes}
 
The first latency stage in the pipeline is the transmission of tiles from the \gls{esp}-CAM to the two worker nodes via \gls{esp}-NOW. Each tile is 96$\times$96, and is 9,216 bytes in size. \gls{esp}-NOW has a maximum frame size of 250 bytes, so each tile is fragmented into 38 sequential packets of 247 data bytes each, with a synchronous ACK-wait between fragments for flow control.
 
Worker~1 was assigned tiles 0–-4 (five tiles) and took 3,243~ms to transmit. Worker~2 was assigned tiles 5–-8 (four tiles) and took 1,183~ms. The difference between these two numbers 3,243~ms vs. 1,183~ms for one more tile is disproportionate and deserves analysis
 
The main reason for this is channel contention. Both workers are on the same 2.4~GHz \gls{esp}-NOW channel and the \gls{esp}-CAM first transmits to Worker~1 then switches to Worker~2. Worker~1 then takes the full serialised cost of $5\times38 = 190$ fragments with ACK-wait on each, but Worker~2 gets a cleaner channel once Worker 1's transmission is over. Worker~1 has a about 648 ms per tile transmission cost (3,243~/~5). Worker~2 has a about 296~ms per tile transmission cost (1,183~/~4). The reason for this 2.2$\times$ difference is the compounding effect of the sequential ACK-wait under a loaded channel, and not an inherent difference in channel quality between the two nodes.
 
Both of these figures are well over the 1,000~ms frame budget target. The transmission stage alone is $4.4\times$ over budget, as the ESP-CAM takes about 4,426 ms (3,243~+~1,183) total time to finish sending all nine tiles. This indicates that the most promising area of optimisation after inference latency is tile fragmentation and sequential ACK-based flow control.
 
\subsection{Worker Node Inference: Tile Extractor and Attention}
 
Each ESP32-S3 worker node runs two TFLite Micro models sequentially per tile: the tile
extractor (60~\gls{kb} INT8, producing a 64-dimensional feature vector) and the
attention head (6~\gls{kb} INT8, producing a scalar logit). Both models execute from a
600~\gls{kb} arena allocated in Octal PSRAM via
\textit{heap\_caps\_aligned\_alloc} with 16-byte alignment.
 
Worker~1 processed five tiles sequentially, completing in 13,284~ms total, yielding an
average per tile inference time of 2,657~ms. Worker~2 processed four tiles sequentially,
completing in 10,627~ms total, yielding an average per tile inference time of 2,657~ms
consistent with Worker~1, confirming that per tile inference time is stable and
node-independent.
 
The dominant contributor to this latency is PSRAM access speed. The tile extractor
requires a minimum arena of 451~\gls{kb}, which exceeds the ESP32-S3's available
internal \gls{sram} ($\sim$180~\gls{kb} free after system allocation). All intermediate
activation tensors therefore reside in Octal PSRAM, which operates at
significantly higher latency than internal \gls{sram} due to the SPI bus overhead
and cache miss penalty. Internal \gls{sram} access on the ESP32-S3 operates at
CPU speed ($\sim$4~ns at 240~MHz), whereas Octal PSRAM access incurs
approximately 20 to 40 clock cycles of latency per access, producing the observed
$\sim$2.66~s per-tile execution time compared to the sub 100~ms that would be expected
for an equivalent model running entirely from \gls{sram}.

\subsection{Feature Transmission: Worker Nodes to Aggregator}
 
Each worker node transmits a 69 byte \textit{WorkerPayload} struct to the aggregator
upon completing inference for each tile. The payload contains the tile ID (1~byte), raw
attention logit (4~bytes as float32), and the quantised 64 dimensional feature vector
(64~bytes as int8). At 69~bytes, this fits comfortably within a single \gls{esp}-NOW
frame (250-byte maximum), requiring no fragmentation.
 
The average one way transmission delay per payload was measured at 472~ms using a
round trip ACK scheme with one way delay estimated as RTT/2. This figure is
substantially higher than the theoretical \gls{esp}-NOW unicast latency of approximately
2 to 5~ms and warrants explanation.
 
The 472~ms figure reflects the queued transmission delay rather than raw channel latency.
Because each worker node processes tiles sequentially and inference takes $\sim$2,657~ms
per tile, payloads accumulate in the transmission queue during inference. The measured
472~ms average therefore includes both the actual \gls{esp}-NOW channel time ($\sim$2 to 5~ms)
and the queuing delay imposed by the serialised inference pipeline. The true channel
transmission time for a 69 byte \gls{esp}-NOW packet is sub millisecond; the observed
472~ms is a system level metric reflecting the pacing of the inference pipeline rather
than a channel limitation.
 
Total transmission time for all nine feature payloads (five from Worker~1 plus four from
Worker~2) is approximately 4,248~ms ($9 \times 472$~ms), though in practice Worker~1
and Worker~2 transmit in parallel once their respective inference completes, reducing
the effective wall clock transmission time to approximately
$\max(5,4) \times 472$~ms $= 2,360$~ms under ideal parallel operation.
 
\subsection{Aggregator Node: Softmax, Weighted Sum, and Classification}
 
The ESP32 aggregator node receives nine \textit{WorkerPayload} packets, performs softmax
normalisation across the nine raw attention logits, computes the attention-weighted sum
of the nine 64-dimensional feature vectors, and passes the resulting aggregated feature
through the INT8 classifier to produce a binary label.
 
The complete aggregator pipeline executed in 0.342~ms, making it the fastest stage in
the entire system by four orders of magnitude. \Cref{tab:aggregator_latency} presents

\subsection{Discussion: Performance Gap and Path to Real-Time Operation}
 
The results show a clear and quantifiable performance gap between the current prototype and the real time target. The system is architecturally correct: all pipeline stages work as intended, data flows correctly between nodes, \gls{int8} inference produces valid outputs, and the aggregator correctly fuses nine partial representations into a binary decision. However, the system currently does not meet the latency requirements to operate in real-time at 1 \gls{fps}.
The results clearly point the way to close this gap. To reduce per tile inference time from 2657 ms to <150~ms, either the tile extractor arena needs to fit entirely within internal \gls{sram} (a model with arena requirement of $<\sim 180$~KB can be achieved by reducing the channel count or by simplifying the architecture), or a hardware platform with faster external memory such as the \gls{esp}32-P4 (with its on chip cache backed PSRAM) needs to be used. Meanwhile, the 3,243~ms Worker~1 transmission bottleneck could be addressed by swapping out the sequential ACK based fragmentation scheme for a broadcast based transmission or by increasing the \gls{esp}-NOW payload limit via custom fragmentation.
The aggregator node, with its 0.342~ms processing time and 4~KB model footprint, confirms that the fusion and classification stage of the architecture imposes essentially zero overhead and would remain viable even as the upstream pipeline is accelerated by one to two orders of magnitude.
